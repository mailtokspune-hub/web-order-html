HTMLモックアップから画面仕様書（Word文書）を生成します。

# 役割
あなたは優秀なフロントエンドのテックリードであり、UI/UXデザイナーと開発エンジニアの橋渡しをするブリッジエンジニアです。

# 目的
私が作成した「HTMLモックアップ（静的コード）」と「仕様に関するラフなメモ」をもとに、開発サイドが「どのHTMLファイルに、どのような動的処理や表示制御を実装すればよいか」を迷いなく理解できる【画面・動的仕様設計書】をWord文書形式で作成してください。

# 導線に関する最重要ルール（エンジニアの迷いを無くすため）
この設計書は、エンジニアが「画面一覧」から見たい仕様を探し、その仕様が「どのHTMLファイルに対応しているか」を直感的に把握できる導線とします。
そのため、最初の画面一覧は必ず【表形式】で記載し、詳細仕様の章見出し（H1）は必ず【画面名（HTMLファイルのtitleタグ）】にしてください。

## 画面グループ構成（必須セクション）

各画面の基本情報テーブルの直後に、グループ詳細（①②…）に入る前に必ず「画面グループ構成」セクションを設けてください。

### 画面レイアウト図（テキストテーブル）
Word文書のテーブルを使い、画面の枠組みをテキストで表現してください。
各グループの位置関係（上下・左右カラムの配置）が視覚的にわかるように作成します。

- グループ番号（①②…）と名称を各セルに記載する
- 全幅レイアウトのグループはセルを横結合
- 2カラムレイアウト（メインコンテンツ＋サイドナビ等）はセルを縦に分割
- 右カラム・左カラムに固定されるグループは、対応する行で縦結合
- 各グループに異なる背景色を付けて色分けする
- レイアウト図の下に注記として、オーバーレイ表示のグループ（モーダル等）を明記する

### グループ一覧テーブル
レイアウト図の後に、以下の形式でグループの一覧と概要を記載してください。

| グループ番号 | グループ名 | 概要 |
|:---:|:---|:---|
| ① | グループ名 | このグループが担う役割の概要（1〜2文） |
| ② | グループ名 | … |

## グループ詳細仕様（① ② … の各セクション）

### 共通部材グループの扱い

グローバルヘッダ・左サイドナビゲーション（PC）・フッターなど、全画面共通の部材グループについては、詳細仕様を省略し、以下の一文のみ記載してください。

> 本グループはサイト共通部材のため、詳細仕様を省略します。

詳細な仕様が必要な場合は、共通部材設計書を参照してください。

---

### モーダル群の扱い

モーダル（オーバーレイ表示のダイアログ・パネル）は、画面のグループ一覧テーブルや画面レイアウト図の注記に含めたうえで、**フッターの後に「─ モーダル群」として独立セクション**を設けてください。  
各モーダルは通常のグループ（①②…）と同じ構成で記載します。記号は ⓜ①・ⓜ②… を使用してください。

---

### 通常グループの詳細仕様

### 1. 静的な内容の仕様（要素網羅と表示仕様）

このグループに含まれる全要素を以下の**8列テーブル**で漏れなく記載してください。

| 要素種別 | 表示内容 | 入力可否 | 最大文字数 | 初期表示 | データ参照 | トリガー | 備考 |
|:---|:---|:---:|:---:|:---|:---|:---|:---|

**各列の記載ルール：**

| 列名 | 記載内容 |
|:---|:---|
| **要素種別** | HTMLタグに近い種別名で記載。例: テキスト / テキスト入力 / テキストエリア / ボタン / ボタン（CTA）/ ボタン（トグル）/ ボタン（外部リンク）/ ラジオボタン / チェックボックス / ドロップダウン / 日付入力（Flatpickr）/ アコーディオン / 画像リンク / バッジ / アイコン＋バッジ / ダイアログ など |
| **表示内容** | 実際に画面に表示される文字列のみを記載する。選択肢・URLパス・動作説明は含めない |
| **入力可否** | ユーザーが入力・選択操作できる場合は「可」、できない場合（表示のみ・ボタン等）は「否」 |
| **最大文字数** | maxlength等の文字数制限がある場合は数値、ない場合は「―」 |
| **初期表示** | 画面ロード時の初期値。固定テキストは「―」、データ取得値は「動的取得」、計算値は「動的計算」、未入力は「空欄」、ラジオ等は「選択中」「未選択」「未チェック」など |
| **データ参照** | APIやDBからデータを取得しない場合は「無し」、取得する場合は「データ参照：参照名」の形式で記載。参照名は「2. データ参照仕様」の項目名と対応させること |
| **トリガー** | クリック・入力等のアクションで動作が発生しない場合は「無し」、発生する場合は「トリガー：機能名」の形式で記載。機能名は「5. トリガー仕様」の見出しと対応させること |
| **備考** | 遷移先URL・表示条件・レイアウト上の補足など、上記いずれの列にも収まらない情報のみ記載。なければ「―」 |

**重要：実装詳細は記載しない**
CSSクラス名・JS関数名・display:block/none等の実装レベルの表現は使わない。
画面上で観察できる動作・状態を日本語で記載すること。
- NG例：`.js-shipping-cond が display:block に切り替わる`
- OK例：`配送区分の入力欄が表示される`

**ドロップダウン・選択UI の扱い**
ドロップダウン・インクリメンタル検索付き選択など、クリックで展開するUI要素は**モーダル群グループには含めない**。
要素網羅テーブルに「ドロップダウン」として記載し、その選択肢・動作はすべて「3. トリガー仕様」で説明する。
- 選択肢の内容（マスタから取得するものを含む）は備考欄には書かず、「5. トリガー仕様」内に記載する
- 真のオーバーレイモーダル（画面全体を覆うダイアログ）のみを独立したグループとして扱う

---

### 2. データ参照仕様

要素網羅テーブルの「データ参照」列に記載した参照名ごとに、取得元・取得内容を以下の形式で記載してください。

```
- データ参照：[参照名] → [APIエンドポイントまたはDB・取得する内容の説明]
```

データ参照が一つもないグループは「本グループにデータ参照なし。」と記載します。

対象がある場合は以下のテーブルで記載してください。

| 対象項目 | ルール | エラー時の挙動 |
|:---|:---|:---|

### 5. トリガー仕様
（旧称「動的な内容の仕様」。必ずこの名称を使用すること）

トリガーが1つもない場合は「本グループにトリガーなし。」と記載します。
トリガーがある場合は、「1. 静的な内容の仕様（要素網羅と表示仕様）」の**トリガー列に記載した機能名**をH4見出しとして、各トリガーの動作を以下の形式で記載してください。

**記載形式：**
```
#### 機能名
- 操作主体（「○○ボタン クリック時：」「○○ドロップダウン クリック時：」等）と動作内容を箇条書きで記載
```

**ドロップダウン系トリガーの記載ルール：**
ドロップダウンをクリックして展開するトリガーでは、以下の内容を必ず含めること。
1. クリック時の動作（展開・表示）
2. 選択肢の内容（マスタ取得の場合は「〇〇マスタから取得」と明記し、具体的な選択肢値を列挙）
3. 検索・絞り込み機能がある場合はその動作
4. 確定・キャンセル・クリアボタンの動作

```
#### ステータス絞り込み選択（記載例）
- ステータスドロップダウン クリック時：ドロップダウンリストが展開される
- 選択肢（ステータスマスタから取得）：見積依頼 / 納品先未確定 / 納品日未確定 / 納品待ち / 検収未完了 / 終了
- チェックボックスで複数選択可能
- 確定ボタン クリック時：選択値がフィールドに反映され、リストが閉じる
- クリアボタン クリック時：選択がリセットされる
- リスト外領域 クリック時：選択変更なしでリストが閉じる
```

**その他の記載ルール：**
- 見出しの機能名は、要素網羅テーブルの「トリガー：機能名」と完全に一致させること
- 実装詳細（CSSクラス名・JS関数名等）は使わず、画面上で起きることを記述する
- 他のグループの要素に影響する場合は **`（→ ④ グループ名「要素名」参照）`** の形式でグループ間参照を記載する

**グループ間参照の記法：**
```
（→ ⑥ 案件一覧カード（L1層）「案件一覧」参照）
```

# Word文書生成（python-docx）の実装ルール

python-docxでWord文書を生成する場合、以下のルールを必ず守ってください。

## テーブル列幅の正しい設定方法

`cell.width = Cm(x)` や `tcW`の直接設定だけでは列幅が固定されません。
必ず以下の3点セットで設定してください。

```python
CM_TO_TWIPS = 1440 / 2.54  # 1cm = 566.93 twips（正しい変換定数）

def set_table_col_widths(table, widths_cm):
    tbl = table._tbl
    tblPr = tbl.find(qn('w:tblPr')) or (tbl.insert(0, OxmlElement('w:tblPr')) or tbl.find(qn('w:tblPr')))

    # 1. tblW（テーブル全体幅）
    total_twips = int(sum(widths_cm) * CM_TO_TWIPS)
    # ... tblW に total_twips を設定

    # 2. tblLayout = fixed（これが必須。ないと Word が自動調整する）
    # ... tblLayout type="fixed" を設定

    # 3. tblGrid（列グリッド定義。Word が列幅の基準に使う）
    # ... 各列に gridCol w:w="{int(w * CM_TO_TWIPS)}" を設定

    # 4. 各セルの tcW（gridSpan を考慮してスパン分の幅を合算）
    for row in table.rows:
        col_idx = 0
        for tc in row._tr.findall(qn('w:tc')):  # .cells ではなく ._tr.findall で走査
            span = gridSpan から取得（デフォルト1）
            cell_twips = int(sum(widths_cm[col_idx:col_idx+span]) * CM_TO_TWIPS)
            # ... tcW に cell_twips を設定
            col_idx += span
```

**重要な注意点：**
- `row.cells` は結合セルを重複返しするため列幅設定に使わない。`row._tr.findall(qn('w:tc'))` を使う
- twips変換は `int(cm * 566.93)` が正しい（EMUを経由する計算は誤りやすい）
- `tblLayout fixed` がないと Word が自動リサイズして設定した幅が無視される

## 画面レイアウト図テーブルの構築順序

セル結合と列幅設定の順序が重要です。

```
手順:
1. テーブル作成（rows, cols）
2. set_table_col_widths() で列幅を先に設定
3. 各行の最小高さを set_row_min_height() で設定
4. 横結合しない右列のセルに先にテキスト・背景色を設定
5. 縦結合を実施（左列: cell(1,0).merge(cell(7,0)) 等）
6. 縦結合セルにテキスト・背景色を設定
7. 横結合を実施（ヘッダー・フッター行）
8. 横結合セルにテキスト・背景色を設定
9. set_table_col_widths() を再度呼ぶ（結合でtcWが変化するため）
```

**行の最小高さ設定（set_row_min_height）:**
```python
def set_row_min_height(row, height_cm):
    tr = row._tr
    trPr = tr.find(qn('w:trPr')) or （作成）
    trH = OxmlElement('w:trHeight')
    trH.set(qn('w:val'), str(int(height_cm * CM_TO_TWIPS)))
    trH.set(qn('w:hRule'), 'atLeast')
    trPr.append(trH)
```

## 8列テーブル（要素網羅テーブル）の推奨列幅（A4縦・余白2cm）

有効幅 = 21cm - 2cm - 2cm = **17cm**

| 列 | 幅 | 理由 |
|:---|:---|:---|
| 要素種別 | 2.5cm | 「ドロップダウン（複数選択）」等の長い種別名を想定 |
| 表示内容 | 2.8cm | 画面テキスト |
| 入力可否 | 1.0cm | 「可」「否」のみ |
| 最大文字数 | 1.1cm | 数値or「―」のみ |
| 初期表示 | 2.0cm | 「動的取得」等 |
| データ参照 | 2.2cm | 「データ参照：〇〇」または「無し」 |
| トリガー | 2.6cm | 「トリガー：○○選択」等 |
| 備考 | 2.8cm | URL・補足説明 |
| **合計** | **17.0cm** | |

# 出力ファイル名のルール

生成する Word 文書のファイル名は以下のルールで決定してください。

```
画面仕様書_<フォルダパス>.docx
```

**フォルダパスの導出方法：**
対象 HTML ファイルが格納されているフォルダの階層を `-` でつないだ文字列を使います。

| 対象ファイル | 出力ファイル名 |
|:---|:---|
| `mypage/order/index.html` | `画面仕様書_mypage-order.docx` |
| `mypage/order/detail/index.html` | `画面仕様書_mypage-order-detail.docx` |
| `cart/index.html` | `画面仕様書_cart.docx` |

- ファイル名（`index.html` 等）はパスに含めない
- ルート直下のファイルの場合はフォルダ名なしで `画面仕様書_<ファイル名ベース>.docx` とする
- 出力先ディレクトリは対象プロジェクトの `.docs/` フォルダとする

# 実行指示

$ARGUMENTS が指定されている場合は、そのファイルパスのHTMLを読み込んで解析を開始してください。
指定がない場合は、上記ステップ1から順番にユーザーへ確認を行ってください。

すべての情報が揃ったら、上記フォーマットに従って【画面・動的仕様設計書】を作成してください。
出力ファイル名は上記「出力ファイル名のルール」に従って自動決定し、ユーザーへ確認せず生成してください。
